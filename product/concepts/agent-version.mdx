## 1. What is an agent version

An **agent version** is one **saved snapshot** of behavior for an agent: model, instructions, schemas, allowed **operations** (tools), safety limits, and optional multi-agent delegation. You need versions so you can **edit and test** without immediately changing production, then **Deploy** exactly one line as **live** for **triggers** and default automation.

### 1.1 Field reference

Settings are grouped in **Version configuration** sub-tabs. Live versions are **read-only** in the UI until you switch to a testing snapshot.

#### Model & Prompt

| Field | What it is | Rules and limitations |
| --- | --- | --- |
| **LLM model** | Model id routed via the agent’s LLM provider. | Must be valid for that provider. |
| **Instructions** | System / task instructions. | Required content per your setup; **Sub-agent** agents also need a **description** (below). |
| **Temperature** | Randomness of completions. | Valid numeric range for the model. |
| **Max iterations** | Cap on model turns in a run. | Enforced at run time. |
| **Max tokens per run** | Token ceiling for the run. | Enforced at run time. |
| **Max tool calls** | Cap on tool invocations per run. | Also limited by **plan**; the editor **clips** to the stricter ceiling on save. |
| **Description** (Sub-agent mode only) | Short text parents see when this agent is a **tool**. | **Required** for Sub-agent agents; empty blocks **Save**. |

#### Input & Output

| Field | What it is | Rules and limitations |
| --- | --- | --- |
| **Input schema** | Expected input shape (e.g. JSON Schema). | **Test run** and production validate input when runs execute. |
| **Output schema** | Expected output shape. | Used for validation / tooling expectations per your configuration. |

#### Operations

| Field | What it is | Rules and limitations |
| --- | --- | --- |
| **Allowed operations** | Checkboxes of **[Integration](/product/concepts/integration)** operations exposed as tools. | Only operations in **your workspace** appear. Empty state directs you to create integrations and operations first. |

![Version configuration — Operations tab, selecting allowed tools](/images/agents-operations-tab.png)

#### Limits & Safety

| Field | What it is | Rules and limitations |
| --- | --- | --- |
| **Anomaly control (L1)** | Toggle for Guard tooling rules on this snapshot. | When on, configure **Rules** (review / block / allow) per tool call; when off, rules UI is hidden. |
| **Max session tokens** | Optional cap on **total** tokens (root + all sub-agent runs) in one **session**. | Blank = no session-wide cap beyond per-run limits. |
| **Max session duration (sec)** | Wall-clock limit from session start. | Exceeding it fails the run with a session duration error. |

#### Multi-agent (plan permitting; Request / HITL agents)

| Field | What it is | Rules and limitations |
| --- | --- | --- |
| **Can execute sub-agents** | Whether this version may delegate to **[Agent](/product/concepts/agent)** agents in **Sub-agent** mode. | Off ⇒ allow-list must be empty; **Execution model** disabled. |
| **Allowed sub-agents** | Which Sub-agent agents may be called. | Only **Sub-agent** agents; cannot add **this** agent to its own list. Plan may cap **count** per version (UI shows limit). |
| **Execution model** | **Sequential** (one sub-agent call per turn) vs **Parallel** (multiple calls to the **same** sub-agent in one turn). | **Parallel** only parallelizes calls to the **same** sub-agent with different inputs; different sub-agents in one message are serialized. Disabled when delegation is off or snapshot is **live**. |

![Version configuration — Multi-agent tab](/images/agent-multi-agent-tab.png)

#### Version metadata

| Field | What it is | Rules and limitations |
| --- | --- | --- |
| **Version label** | Human-readable id (e.g. `v1.2`). | **Unique per agent**; duplicates fail on **Save**. |

![Version configuration — tabs, version selector, Save and Deploy](/images/agent-version-details.png)

## 2. How to create an agent version

New snapshots are usually created from the **version** menu on **Version configuration** (**Add** / duplicate flows as your Phrony build exposes). After creation, edit fields on a **testing** snapshot and **Save**.

## 3. How to update an agent version

1. Select a **testing** snapshot in the version dropdown.
2. Change fields in **Model & Prompt**, **Input & Output**, **Operations**, **Limits & Safety**, or **Multi-agent**.
3. **Save**. **Live** snapshots cannot be edited in place—work on testing, then **Deploy**.

Use **Rollback** to make an older testing snapshot **live** again.

## 4. How to delete an agent version

From the version menu, **delete** only snapshots that are **not** live. The **live** line must be replaced by deploying another version before you can remove it.

## 5. Permissions

| Action | Owner | Admin | Developer | Operator |
| --- |:---:|:---:|:---:|:---:|
| Open **Version configuration** | Yes | Yes | Yes | **No** |
| Edit **testing** version | Yes | Yes | Yes | **No** |
| **Deploy** / **Rollback** | Yes | Yes | Yes | **No** |
| **Delete** testing version | Yes | Yes | Yes | **No** |

## 6. Limitations

- **Plan**: sub-agent **depth**, **multi-agent** feature flag, and **max tool calls** cap what you can save or run.
- **Live** line is unique; automation always uses **live** for triggers (see **[Run](/product/concepts/run)**).

## 7. Notes

- **Test run** can target whichever snapshot is selected (**testing** or **live**).
- New versions get defaults (e.g. anomaly off, delegation off, Sequential model) until you change them.

## 8. Cautions

- Invalid **Guard** rule sets can fail **Save** with builder messages.
- Delegation mistakes (depth, cycles) may surface at **run time** even when **Save** succeeded.

## 9. Common errors

| Symptom | Typical cause |
| --- | --- |
| **Save** fails on label | Duplicate **version label** for this agent. |
| Fields greyed out | You are on the **live** snapshot; switch to **testing** or deploy a new line. |
| Cannot add sub-agent | Target not in **Sub-agent** mode, self-reference, over plan count, or multi-agent not on plan. |
| Runs hit tool cap sooner than field | **Plan** tool-call cap is stricter than the number in the form. |
