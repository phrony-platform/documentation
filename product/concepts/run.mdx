## 1. What is a run

A **run** is **one execution** of an **[agent version](/product/concepts/agent-version)** against an **input**: model steps, **operation** calls, optional pauses for human review, and a **final status**. Runs belong to a **[session](/product/concepts/session)**; the **root** run is what you usually open from session tables, with **child** runs for **sub-agent** delegations.

### 1.1 Field reference

In **Run inspector** (and related API views), you typically see:

| Field / area | What it is | Rules and limitations |
| --- | --- | --- |
| **Status** | Running, succeeded, failed, waiting on human, etc. | Drives badges and whether actions are allowed. |
| **Timing** | Start, end, duration. | Running runs have open-ended timing. |
| **Tokens** | Usage attributed to this run. | Child runs have their own totals; session totals aggregate. |
| **Agent version** | Which snapshot executed. | **Test run** uses the **selected** snapshot; **triggers** use **live** only. |
| **Triggered by** | **Test run**, named trigger, schedule, event, etc. | Matches session list **Triggered by** context. |
| **Input** | Payload validated against **Input schema**. | Bad input fails early with validation errors. |
| **Trace / timeline** | Model and **operation** steps. | Layout follows **Settings → General → Session trace layout** (**Basic** vs **Advanced**). |
| **Conversation / I/O** | Structured views of messages or I/O. | Depends on agent configuration. |
| **Sub-agent sections** | Child runs when delegation occurred. | **Parallel** same-sub-agent batches may show grouped behavior by release. |
| **Pause / review** | HITL or Guard panels. | Appears when the run waits on a person. |

## 2. How to create a run

- **Test run**: In **[Agent](/product/concepts/agent)** workspace, **Version configuration** or **Sessions**, choose **Test run**, provide **Visual** or **JSON** input, run. Uses whichever version is **selected** in the dropdown.
- **Automation**: **[Trigger](/product/concepts/trigger)** (schedule, API, event) or HTTP API. Always uses the **live** deployed version—see §3 vs §4 consistency in **[Limits & plans](/product/reference/platform-rules)**.

## 3. How to update a run

Runs are immutable execution records. “Update” in practice means **human actions** on pauses (**Approvals**, in-run prompts) or **continuing** a session from the dashboard—not editing the run log after the fact.

## 4. How to delete a run

There is no user-facing **delete run** in the dashboard; retention is governed by your Phrony deployment and policy.

## 5. Permissions

| Action | Owner | Admin | Developer | Operator |
| --- |:---:|:---:|:---:|:---:|
| **Test run** | Yes | Yes | Yes | **No** |
| Open **Run inspector** from agent workspace | Yes | Yes | Yes | **No** |
| Open run from **Approvals** task | Yes | Yes | Yes | Yes (typical review path) |

## 6. Limitations

- **Max operation calls**, per-run token caps, and session caps on the **version** still apply.
- **Plan** may further cap operation calls (stricter of plan vs version wins).

## 7. Notes

- Child runs share the parent **session** and contribute to **session** token totals and limits.
- **API** triggers may show rate / IP controls on the trigger—see **[Trigger](/product/concepts/trigger)**.

## 8. Cautions

- Assuming **Test run** uses **live** is wrong—always check the **version** dropdown.
- **Terminate session** (from **Dashboard** home) fails **all in-progress runs** in that session.

## 9. Common errors

| Symptom | Typical cause |
| --- | --- |
| Run will not start | Agent **blocked** by Guard, trigger **inactive**, no **live** version for automation, invalid provider/secret, or built-in integration/hosting issue—see **[Limits & plans](/product/reference/platform-rules)**. |
| Input rejected | Does not satisfy **Input schema**. |
| Operation cap errors | **Plan** or version **Max operation calls** exceeded. |
