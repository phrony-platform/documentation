## 1. What is a session

A **session** is the container for one thread of work: it groups the **root [run](/product/concepts/run)** and any **child runs** from **[sub-agent](/product/concepts/agent-version)** delegation, shared **status**, **total token usage**, and **how work started** (for example **Test run**, **schedule**, **API**, or **event**). You need sessions so usage, billing, and limits can apply to the **whole conversation**, not a single run.

Sessions are **not** a separate object you create empty—they open when a run starts.

### 1.1 Field reference

What you see in dashboard tables (**Dashboard** home **Recent sessions**, or an agent’s **Sessions** tab) is **read-only** projection:

| Field / column | What it is | Rules and limitations |
| --- | --- | --- |
| **Session** (id) | Short identifier for the row. | Stable for that session; use for support and cross-links. |
| **Agent** | Which agent owns the session. | Shown on tenant-wide lists. |
| **Trigger type** | **API**, **Manual**, **Scheduled**, **Event**, etc. | Derived from how the root run started. |
| **Triggered by** | Human-readable source (trigger name, key label, etc.). | May be empty in edge cases. |
| **Status** | **Running**, **Completed**, **Failed**, **Awaiting approval**, **Awaiting reply**, … | Maps internal states to dashboard labels. |
| **Tokens** | Aggregate usage for the session. | Includes root and sub-agent runs where applicable. |
| **Version** | Label of the **[agent version](/product/concepts/agent-version)** used. | Agent workspace **Sessions** can filter **This snapshot only**. |
| **Started / Finished** | Timestamps. | **Running** sessions may have no finish time yet. |

Optional **session-wide** caps (**Max session tokens**, **Max session duration**) are configured on the **version**, not on the session row.

## 2. How to create a session

Not applicable as a manual step. A session is **created automatically** when a run starts (**Test run**, **[Trigger](/product/concepts/trigger)**, or HTTP API).

## 3. How to update a session

There is no generic “edit session” form. Status changes come from **run progress**, **operator** actions (**Mark complete** where offered), or **[Approvals](/product/dashboard/approvals)** resolutions.

## 4. How to delete a session

There is no “delete session” button. You can **terminate** an in-progress session from **Dashboard** home **Recent sessions** when the UI offers **Terminate session** (fails in-flight runs; cannot be undone—see **Cautions**).

## 5. Permissions

| Action | Owner | Admin | Developer | Operator |
| --- |:---:|:---:|:---:|:---:|
| View **Recent sessions** (Dashboard home) | Yes | Yes | Yes | **No** |
| View **Sessions** on an agent | Yes | Yes | Yes | **No** |
| **Mark complete** / **Terminate** (when shown) | Yes | Yes | Yes | **No** |

Operators interact with work through **[Approvals](/product/dashboard/approvals)** links, not the full session list.

## 6. Limitations

- **[Agent version](/product/concepts/agent-version)** may define **Max session tokens** and **Max session duration**; exceeding them fails the run.
- **Monthly session** allowance and **plan inventory** can block **new** sessions—see **[Limits & plans](/product/reference/platform-rules)**.

## 7. Notes

- Parallel child runs (same sub-agent, one parent turn) still roll up into **one** session’s token totals.
- **Dashboard** home may show a **plan compliance** banner when the workspace is over limits after a plan change.

## 8. Cautions

- **Terminate session** is destructive for in-progress work; confirm you intend to fail active runs.

## 9. Common errors

| Symptom | Typical cause |
| --- | --- |
| New work will not start | Monthly session cap, inventory cap, or post–plan-change overage—see **[Limits & plans](/product/reference/platform-rules)**. |
| Session hits token/time ceiling | Version **session** limits exceeded; raise caps or reduce work per session. |
| Row has no link to run | Rare edge case where the UI cannot resolve a root run link. |
