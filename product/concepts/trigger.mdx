## 1. What is a trigger

A **trigger** starts an **[agent](/product/concepts/agent)** outside **Test run**: on a **cron schedule**, via an **HTTP API** surface, or when an **event** arrives from a **[built-in integration](/product/concepts/integration)**. Triggers connect automation to the **live** version line.

### 1.1 Field reference

| Field / concept | What it is | Rules and limitations |
| --- | --- | --- |
| **Name** | Label in lists and **Triggered by**. | **Unique per agent**. Duplicates fail on save. |
| **Type** | **Scheduled**, **API**, or **Event**. | **Sub-agent** agents: **Triggers** tab **hidden**—no triggers. |
| **Active** | On/off switch. | **Off** ⇒ nothing fires; failures often cite inactive trigger. |
| **Schedule + timezone** (scheduled) | Cron and IANA timezone. | Invalid or empty schedule rejected on save. |
| **Payload** (scheduled) | Optional static JSON as run input. | Must be valid JSON when provided. |
| **Rate limits, IP allowlist** (API) | Optional gateway limits. | **Only** for **API** type; wrong type + filled fields may fail validation. |
| **Integration + event** (event) | Built-in service and event type. | Requires **connected** built-in integration; wizard steps **Integration → Event type → Event configuration**. |
| **Concurrency / token caps** (API, when offered) | Optional throttles on API path. | See your Phrony build’s API trigger form. |

**At most one** **API** trigger per agent; the type picker may show **API (already exists)** when one is present.

## 2. How to create a trigger

1. Open the agent (**Owner**, **Admin**, **Developer**); ensure it is **not** **Sub-agent** mode.
2. Go to **Triggers** → start **New trigger** (or equivalent).
3. Wizard: **Name & type**, then **Schedule & payload** (scheduled), **API limits** (API), or **Integration / Event type / Event configuration** (event).
4. Save / **Create** on the last step.

## 3. How to update a trigger

Open **Triggers**, select the row, edit fields (schedule, active flag, limits, etc.), save. **API** singleton must be edited, not duplicated.

## 4. How to delete a trigger

Use the trigger row’s **delete** action (wording depends on build); confirm in the dialog. Removing a trigger does not delete historical sessions.

## 5. Permissions

| Action | Owner | Admin | Developer | Operator |
| --- |:---:|:---:|:---:|:---:|
| View / create / edit / delete triggers | Yes | Yes | Yes | **No** |

## 6. Limitations

- Automation always uses the **live** **[agent version](/product/concepts/agent-version)**; if nothing is deployed, starts fail until **Deploy**.
- **Event** triggers require a healthy **built-in** integration and correct hosting.
- **Plan** may cap **scheduled triggers** count workspace-wide—see **[Limits & plans](/product/reference/platform-rules)**.

## 7. Notes

- **Test run** is separate: it can run a **testing** snapshot from the dropdown.
- Cron runs in the timezone you set on the trigger.

## 8. Cautions

- Rotating API keys or secrets used by event delivery may break triggers until updated.
- IP allowlists can block legitimate callers if misconfigured.

## 9. Common errors

| Symptom | Typical cause |
| --- | --- |
| Second **API** trigger rejected | Only one API trigger allowed per agent. |
| Event never fires | Integration disconnected, inactive trigger, or hosting/event config gap. |
| Save fails on limits | Limits filled on non-API type or invalid numeric fields. |
| Run starts on wrong behavior | **Live** version not what you expected—deploy the intended snapshot. |
