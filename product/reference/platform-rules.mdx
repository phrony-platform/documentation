Use this page when a button is missing, **Save** or **Deploy** fails, or a run will not start—you’ll find **workspace roles**, **plan limits**, and **what to try next** in plain language.

## Workspace roles in the dashboard

| What you’re trying to do | Owner | Admin | Developer | Operator |
|--------------------------|:-----:|:-----:|:---------:|:--------:|
| Open **Dashboard** home / analytics | Yes | Yes | Yes | **No** |
| View **Secret Manager** (names, kinds) | Yes | Yes | Yes | **No** |
| **Add** or **remove** secrets | Yes | Yes | No | No |
| View **LLM Providers** | Yes | Yes | Yes | **No** |
| **Add** or **edit** LLM providers | Yes | Yes | No | No |
| Work with **Agents**, **Integrations**, runs, triggers | Yes | Yes | Yes | **No** |
| **Guard** (beta), when enabled | Yes | Yes | Yes | Yes |
| **Settings**: profile, API keys, members, billing | Varies by tab | Varies | Varies | Usually **approvals / review** only |

**Operator** is meant for people who **approve** or **review** work (for example Guard), not for building agents. The main builder items above are **hidden** for that role.

Inside **Settings**, tabs such as **Members** or **Billing** skew toward **Owner** and **Admin**; Developers see what your workspace allows.

## Agent and version behavior

- **Agent names** must be **unique** in the workspace—duplicate names are rejected when you save.
- **Live (deployed) versions** cannot be edited in the form; fields are **disabled** until you switch to a **testing** snapshot or deploy a new one.
- You **cannot delete** an agent while a **live** version still exists—the delete dialog explains this.
- **Sub-agent mode** checks your plan’s **cap** on how many sub-agent agents you may have; hitting the cap shows an **upgrade** message.

## When Phrony blocks every new run on an agent

**Guard / Anomaly Control** can mark an agent **blocked until reviewed**. While blocked, **Test run**, **triggers**, and external automation all refuse to start. Someone with the right role must **clear the block** after review. The UI may show a **reason** or **incident** reference.

## Monthly sessions and plan inventory

When a **new session** starts, Phrony checks your **plan** and **current usage**:

- If you are **over** your included **sessions for the month** and your plan **does not allow overage**, you see a **limit reached** message and may need to **upgrade** or wait for the next period.
- If **overage** is allowed, the session may still start and may be **labeled** differently in usage views.
- Your plan can also cap **how many** agents, LLM providers, **scheduled triggers**, or **custom connectors** you may create. Hitting a cap blocks **creating** more until you **upgrade** or remove unused items.

## Tool calls during a run

The **maximum tool calls** you set on a version is **further limited** by your **plan**. Phrony uses the **stricter** of the two. The version editor already **clips** the number you type to your plan’s ceiling when you save.

## Sub-agent delegation: Sequential vs Parallel

On **Version configuration → Multi-agent**, when **Can execute sub-agents** is on, **Execution model** controls how the **parent** model may call sub-agents **in one turn**:

- **Sequential — one sub-agent per turn** — at most one sub-agent invocation per model turn.
- **Parallel — same sub-agent, multiple calls in one turn** — several calls in the **same** assistant message only if they all target the **same** allowed sub-agent (with different inputs); those child runs may run **at the same time**. Calls to **different** sub-agents in the same message are **serialized** (one after another).

Your plan may also set a **maximum delegation depth**; chains deeper than that are blocked when Phrony tries to start the next child run.

## LLM provider must be complete

Runs fail if the agent’s provider has **no API key** selected, the secret was deleted, or the key is invalid. Fix **LLM Providers** and **Secret Manager**, then try again.

## Built-in integrations and hosting

If a version uses **built-in** connector actions, your organization’s **Phrony hosting** must include the right platform configuration. If that is missing, runs fail at **start** with a message about built-in integrations—even though **Save** on the version might have succeeded. Your platform team resolves this outside the dashboard.

## Test run vs triggers (which version runs)

| How it starts | Which snapshot runs |
|----------------|----------------------|
| **Test run** (button in the agent workspace) | Whatever version is **selected** in the dropdown (**testing** or **live**) |
| **Schedule** | Always the **live (deployed)** version |
| **API** trigger | Always the **live** version |
| **Event** trigger | Always the **live** version |

Until you **Deploy** at least once, **triggers** have no live line to run.

## Secrets and providers

- Secret **names** are unique per workspace; duplicates are rejected when you add one.
- There is **no** screen to **change** a secret’s value—add a new secret and **re-select** it on providers or integrations, then remove the old secret if unused.
- Provider **names** are unique; **Self-hosted** needs a reachable **base URL**.

## Version actions at a glance

| Action | Live (deployed) version | Testing version |
|--------|-------------------------|-----------------|
| Edit fields in **Version configuration** | **No** (disabled) | **Yes** |
| **Delete** from version menu | **No** | **Yes** |
| **Deploy** | Already live—no change | Becomes live; others become testing |
