## What “embedded” means here

An **embedded agent** is still a normal **[agent](/product/concepts/agent)** in Phrony. You **design and govern it in the Phrony dashboard**; your **own application or service starts runs** and passes context (for example user id, case id, or a structured payload). The model, instructions, allowed **[integrations](/product/concepts/integration)**, **[human-in-the-loop](/product/hitl/overview)** gates, and **[multi-agent](/product/multi-agent/overview)** behavior are the same as for any other deployment path.

Embedding is a **calling pattern**, not a separate product mode: one agent definition, **multiple surfaces** (web app, mobile backend, ops console, batch jobs), each invoking the **[HTTP API](/api-reference/introduction)** or other **[triggers](/product/concepts/trigger)** you configure.

## Why teams embed agents

- **Decisions where work happens** — Users or internal staff stay inside your product; the agent reasons and acts through Phrony **[runs](/product/concepts/run)** instead of copying context into a separate chat tool.
- **One source of truth** — **[Sessions](/product/concepts/session)** and run history stay in Phrony, so support, compliance, and engineering share the same trace regardless of which client started the run.
- **Consistent guardrails** — Approval rules, integration scoping, and workspace **[limits](/product/reference/platform-rules)** apply on every call path, not only on runs started from the dashboard.

## How invocation fits together

1. **Build the agent** — **[Agent version](/product/concepts/agent-version)** (prompts, tools, caps), **deploy** the line you want automation to use.
2. **Expose a start path** — Most product embeds use an **API** **[trigger](/product/concepts/trigger)** (one per agent) so your servers can start runs with a JSON payload. **Scheduled** and **event** triggers are useful for batch or integration-driven workloads in the same agent.
3. **Call from your stack** — Your backend (or approved automation) calls Phrony with **API keys** from **Settings**, following your org’s HTTP contract. Day-to-day operators still use the **Phrony dashboard** for configuration and review.

**Sub-agent** agents are not API-started on their own; a parent agent invokes them as delegated work. For a user-facing feature, the embeddable unit is typically a **Request** or **HITL** root agent, with **[multi-agent](/product/multi-agent/overview)** delegation configured on its **[version](/product/concepts/agent-version)** if needed.

## Human and agent in the loop

- **HITL** — If the root agent uses **HITL** **execution mode**, **[approval-gated operations](/product/hitl/gating-operations)** pause runs the same way whether the run was started from your app or from **Test run**. Operators resolve items in **[Approvals](/product/dashboard/approvals)** or the run inspector.
- **AITL** — **[Agent in the loop](/product/agent-in-the-loop/overview)** coordination (parent answering a **Sub-agent** question) is unchanged. If the session root is **HITL**, a parent can **[escalate to a human](/product/agent-in-the-loop/human-escalation)**; that path is the same for embedded runs.

## Practical constraints

- Automation always follows the **live** **agent version** until you **deploy** a different snapshot—plan releases accordingly.
- **[Guard & anomaly](/product/dashboard/guard-anomaly)** and plan caps can **block** starts; see **[platform rules](/product/reference/platform-rules)**.
- **API** triggers support **rate limits** and (when your build offers it) **IP allowlists**—treat them as part of your public or partner-facing surface.
