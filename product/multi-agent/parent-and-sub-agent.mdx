## Parent vs Sub-agent

| Role | Agent **execution mode** | How it runs |
| --- | --- | --- |
| **Parent** | **Request** or **HITL** | Started by **Test run**, **[triggers](/product/concepts/trigger)**, or API. Owns **[sessions](/product/concepts/session)** for delegated work. |
| **Child (Sub-agent)** | **Sub-agent** | **Not** started on its own—no **Triggers** tab. Invoked only when a parent’s model invokes the child via a **delegated operation**. |

Each agent has its own **[LLM provider](/product/concepts/llm-provider)** and **[agent versions](/product/concepts/agent-version)**. The child’s **model**, **instructions**, **input schema**, and **operations** are independent of the parent’s.

More: **[Agent](/product/concepts/agent)**.

## Sub-agent-only fields

On a **Sub-agent**’s **Version configuration → Model & Prompt**:

- **Description** — **Required**. Short text the **parent’s** model sees when the child is exposed for delegation (name + summary). Empty **Description** blocks **Save**.
- **Instructions** — Task prompt for the child run, same as any agent.

Parents do **not** use the Sub-agent **Description** field; they use normal **Instructions** plus the delegated **operations** the platform injects for allow-listed children.

## Multi-agent tab (parent version)

On a **Request** or **HITL** parent’s **testing** snapshot:

1. **Can execute sub-agents** — master switch for delegation on this version.
2. **Allowed sub-agents** — which **Sub-agent** mode agents may be called. You cannot add the parent to its own list. The UI enforces a **per-version** allow-list **count** from your **plan**.
3. **Execution model** — **Sequential** or **Parallel** (see **[Sequential and Parallel](/product/multi-agent/sequential-and-parallel)**).

The **Multi-agent** tab is **hidden** for agents in **Sub-agent** mode and may be **hidden** if multi-agent is not on your plan.

Full column reference: **[Agent version](/product/concepts/agent-version)** → Multi-agent.

## Live line for children

When a parent delegates at run time, Phrony runs the child’s **live (deployed)** **[agent version](/product/concepts/agent-version)**—not an arbitrary testing snapshot. **Test run** on the parent still uses whichever snapshot is **selected** in the dropdown, but each child invocation uses that child agent’s **live** configuration.

Implication: **Deploy** Sub-agent versions you expect parents to call in production before relying on delegation in **live** parent traffic. See **[Limits & plans](/product/reference/platform-rules)** → Test run vs triggers.

## HITL parents and children

A **HITL** parent can pause for **[Approvals](/product/dashboard/approvals)** or **Run** review on its own **operation** calls (including those that gate integration **operations**). Child runs may also pause if the **child** version uses **HITL**-relevant behavior (for example **Require approval for this operation** on **[integrations](/product/concepts/integration)**). **[Run](/product/concepts/run)** shows **Sub-agent** sections and pause states per run.

## Permissions and operators

Builders (**Owner**, **Admin**, **Developer**) edit parents and Sub-agents. **Operators** generally do not open **Agents**; they act on approval tasks linked from runs.
