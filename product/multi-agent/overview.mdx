**Multi-agent** in Phrony means a **parent** **[agent](/product/concepts/agent)** (in **Request** or **HITL** mode) delegates work to one or more **child** agents configured in **Sub-agent** mode. The parent’s **[agent version](/product/concepts/agent-version)** exposes those children as **delegated operations** the model can invoke; each invocation runs a **child [run](/product/concepts/run)** inside the same **[session](/product/concepts/session)** as the parent.

## Why use it

- **Specialization** — narrow **instructions**, **schemas**, and **operations** per child (research, drafting, operation-heavy steps) while the parent coordinates.
- **Reuse** — the same Sub-agent can be allow-listed on several parent versions.
- **Throughput** — with **Parallel** execution, multiple calls to the **same** Sub-agent in one parent turn can run together (see **[Sequential and Parallel](/product/multi-agent/sequential-and-parallel)**).

## What multi-agent is not

- **Not** the same as **Request / HITL / Sub-agent** on the **agent** row—that is **execution mode** (who may start the agent). **Sequential / Parallel** on the version is **execution model** (how sub-agent **calls** are batched in a turn). **[Agent](/product/concepts/agent)** → Cautions.
- **Not** a separate product: configuration lives on **Version configuration → Multi-agent** and on each **Sub-agent**’s version (prompt, description, allowed **operations**).

## Who can use it

**Owner**, **Admin**, and **Developer** configure agents and versions. **Operators** typically see delegation only through **[Approvals](/product/dashboard/approvals)** when a parent or child run pauses. Plans may gate **multi-agent**, **sub-agent count**, and **depth**—see **[Limits and troubleshooting](/product/multi-agent/limits-and-troubleshooting)**.
