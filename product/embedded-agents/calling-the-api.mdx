The Phrony HTTP API gateway is **`https://api.phrony.com`**. Use that host for the examples below unless your workspace uses a **dedicated API host** (your **API** trigger screen and admin will match your environment). Replace **`AGENT_UUID`** and the API key with values from the Phrony dashboard.

## Runs on the gateway (not session URLs)

On **`api.phrony.com`**, **starting** automation is always **`POST /v1/agents/{agentId}/runs`**. There is no public **`POST .../sessions`** on that host for kicking off work.

- **`GET /v1/agents/{agentId}/sessions`** — optional **list** of **[sessions](/product/concepts/session)** (pagination query params), not a start endpoint.
- **`/v1/runs/{runId}/...`** — status, stream, conversation, and follow-up **input** are all keyed by **`runId`** after the start response.

## Before you call

1. **Agent** — Not **Sub-agent** mode; a **live** **[agent version](/product/concepts/agent-version)** is **deployed**.
2. **API trigger** — Exactly one **API** **[trigger](/product/concepts/trigger)** on that agent (required for this path).
3. **API key** — Under **Settings**, create a key whose scope includes this **agent** and **API** trigger. The raw value looks like `phk_…` and is shown only once—store it as a secret in your backend.

The gateway authenticates with the **`X-API-Key`** header. The key must match the **`agentId`** in the URL.

## Start a run

**Request**

- **Method / path:** `POST /v1/agents/{agentId}/runs`
- **Headers:** `Content-Type: application/json`, `X-API-Key: phk_…`
- **Body:** JSON with an optional **`input`** object. Omit `input` or send `{}` if the run needs no structured payload. The object is what your **[version](/product/concepts/agent-version)** input schema (if any) expects.

**Example (curl)**

```bash
curl -sS -X POST 'https://api.phrony.com/v1/agents/AGENT_UUID/runs' \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: phk_your_secret_key' \
  -d '{
    "input": {
      "userId": "usr_123",
      "caseId": "case_456",
      "message": "Summarize the latest notes and draft a reply."
    }
  }'
```

Typical success response is **`202 Accepted`** with JSON that includes **`runId`** (and often **`sessionId`** for dashboard correlation). Treat **`runId`** as the handle for **`GET /POST https://api.phrony.com/v1/runs/{runId}/...`**; do not switch to session-scoped paths for the public gateway.

## After the run starts

From your backend you can drive or observe the same run the dashboard would show, using the **same API key** when your key scope matches the run’s agent and trigger. Use the same base URL as for **`POST /v1/agents/.../runs`** (for most workspaces, **`https://api.phrony.com`**).

| Action | HTTP |
| --- | --- |
| Run status | `GET https://api.phrony.com/v1/runs/{runId}` |
| Conversation / trace payload | `GET https://api.phrony.com/v1/runs/{runId}/conversation` |
| Stream updates (SSE) | `GET https://api.phrony.com/v1/runs/{runId}/stream` |
| Send user input into a waiting run | `POST https://api.phrony.com/v1/runs/{runId}/input` (body per your contract; **`/messages`** is an equivalent path on the gateway) |

### Conversation: root run, child runs, and `runs`

**`GET …/v1/runs/{runId}/conversation`** returns a **session-level** payload, not only the single run you put in the path. The gateway resolves that **`runId`** to its **[session](/product/concepts/session)** (root or **[child run](/product/multi-agent/sessions-and-runs)**—either works).

- **`runs`** — Every **[run](/product/concepts/run)** in the session (root and **Sub-agent** children), with **`parentRunId`**, **`depth`**, **status**, and timing. Use this list to discover child run ids and which run is waiting.
- **`items`** — Merged timeline of steps; each item includes **`runId`**. Sub-agent delegation steps may include a child run id inside **`content`** (for example `child_run_id` / `childRunId`), depending on step shape.
- **`session`** — Session metadata for the same thread.

**`GET …/v1/runs/{runId}/stream`** is also scoped to that **session** (real-time updates for the whole thread). For follow-up **`POST …/input`**, target the **`runId`** of the run that is actually waiting (often the root).

Paths and bodies for **`/v1/runs/...`** are also summarized on the **API** trigger detail view in the dashboard (for copy/paste). Listing sessions remains **`GET /v1/agents/{agentId}/sessions`** only—see **Runs on the gateway** above.

## Errors to expect

| Situation | Typical outcome |
| --- | --- |
| Missing or wrong **`X-API-Key`** | **401** — key invalid or not scoped to this agent / API trigger |
| Agent blocked by **Guard** | **409** with an anomaly / block code — see **[Guard & anomaly](/product/dashboard/guard-anomaly)** |
| Rate or concurrency limits on the **API** trigger | **429** or similar — adjust limits or backoff |
| Wrong **`agentId`** in the path | **403** — key not valid for that agent |
