Phrony is the fastest way to take an agent from prototype to production. You build the agent. Phrony handles everything around it — tool access, human approvals, execution history, safety checks, and the runtime itself.

It is not a chat interface. It is not a workflow engine — it does not model fixed sequences of steps, and it does not encode logic upfront. Phrony sits a layer below: the infrastructure your agent needs to actually run in production, so you don't have to build it yourself.

You define what your agent does. Phrony defines how it is allowed to operate — and runs it.

What you get out of the box:

- Governed tool access — every integration call passes through approval rules and inline safety checks before it executes
- Full execution history — every reasoning step, tool call, and decision is recorded, inspectable, and replayable
- Structure without rigidity — agents decide their own path, inside the session limits, depth limits, and safety budgets you set
- Guardrails by default — what an agent can access, how it acts, and how it handles data are enforced by the runtime, not the prompt

Stop building agent infrastructure. Ship the agent.

If you are deciding between a workflow and an agent — or whether Phrony fits your use case — read **[When to use Phrony](/when-to-use-phrony)**.
