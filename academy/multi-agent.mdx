This guide sets up one **parent** **[agent](/product/concepts/agent)** (**Request** or **HITL** mode) that can delegate to other agents running in **Sub-agent** mode. Read the **[Multi-agent](/product/multi-agent/overview)** documentation section for concepts; field tables remain on **[Agent version](/product/concepts/agent-version)** → **Multi-agent**.

**Prerequisites**

- Your **plan** includes **multi-agent** (and any **sub-agent depth** cap your org uses).
- You can edit **Agents** and **Version configuration** (**Owner**, **Admin**, or **Developer**).

---

## Step 1 — Create Sub-agent agent(s)

1. **Agents** → **New Agent** → set **execution mode** to **Sub-agent**.
2. Attach an **[LLM provider](/product/concepts/llm-provider)** (same as any agent).
3. Open **Version configuration** on a **testing** snapshot.
4. **Model & Prompt:** fill **instructions** and, for **Sub-agent** mode, the required **description** (short summary for parent agents).
5. Configure **Operations**, schemas, and limits as needed for the child’s job.
6. **Save**.

Sub-agents **do not** show a **Triggers** tab; parents invoke them as delegated **operations**. More: **[Agent](/product/concepts/agent)**.

---

## Step 2 — Publish child line(s) you trust

1. **Test run** the Sub-agent version until behavior is correct.
2. **Deploy** the version you want parents to call in production (**live** line).

Parents always use the child’s **live** configuration when delegation runs in real traffic (consistent with **[Run](/product/concepts/run)** and triggers using **live**).

---

## Step 3 — Configure the parent agent

1. Create or open a **Request** or **HITL** agent (not Sub-agent).
2. **Version configuration** → **Multi-agent** (tab appears when your plan supports it).
3. Turn on **Can execute sub-agents**.
4. Under **Allowed sub-agents**, select one or more **Sub-agent** agents (respect the plan’s **per-version** cap).
5. Choose **Execution model**:
   - **Sequential — one sub-agent per turn** — at most one child call per model turn.
   - **Parallel — same sub-agent, multiple calls in one turn** — multiple simultaneous child runs only when every call targets the **same** sub-agent with different inputs; different sub-agents in one message are **serialized**.

6. **Save** on a **testing** snapshot.

![Version configuration — Multi-agent tab, allowed sub-agents, execution model](/images/agent-multi-agent-tab.png)

More: **[Agent version](/product/concepts/agent-version)** → **Multi-agent**.

### Example: **Parallel** — same sub-agent, three lookups in one turn

**Parallel** only batches multiple calls when each call targets the **same** Sub-agent with **different** inputs in the **same** parent turn. Below is one coherent lab: one child **Company brief** agent, one parent **Research coordinator** that fans out one call per company.

**1. Sub-agent (e.g. name `Company brief`)**

- **Description** (required for Sub-agent; parents see this as the delegation summary):

```text
Given `company` and `focus`, returns a tight factual paragraph (max ~4 sentences). One company per call.
```

- **Instructions:**

```text
You are a company research micro-agent. Input is JSON matching the input schema: `company` (string) and `focus` (string).

Write one short paragraph for executives: what matters about `company` regarding `focus`. Be specific; if you are unsure, say so briefly. No bullet lists unless the input asks for them.
```

- **Input schema** (child):

```json
{
  "type": "object",
  "properties": {
    "company": { "type": "string", "description": "Company name" },
    "focus": { "type": "string", "description": "Angle to research (e.g. AI strategy)" }
  },
  "required": ["company", "focus"]
}
```

Enable whatever **operations** or defaults the child needs; **Save** and **Deploy** the child **live** (Step 2).

**2. Parent agent — Multi-agent tab**

- **Allowed sub-agents:** `Company brief` (your Sub-agent).
- **Execution model:** **Parallel — same sub-agent, multiple calls in one turn**.

**3. Parent — Model & Prompt**

- **Instructions:**

```text
You are a research coordinator. The user supplies `companies` (comma-separated names) and `focus`.

You must delegate to the Company brief sub-agent: exactly one **operation** call per company name, each with input `{ "company": "<single name>", "focus": "<user's focus>" }`. Do not merge multiple companies into one call—that breaks Parallel fan-out.

After all sub-agent results are back in this turn, reply with a markdown table:

| Company | Summary |

Use one row per company, using only the sub-agent text for that row.
```

- **Input schema** (parent):

```json
{
  "type": "object",
  "properties": {
    "companies": {
      "type": "string",
      "description": "Comma-separated company names, e.g. Acme Corp, Globex, Initech"
    },
    "focus": { "type": "string", "description": "Research angle shared across all calls" }
  },
  "required": ["companies", "focus"]
}
```

**Save** the parent testing snapshot.

**4. Parent — Test run (JSON)**

```json
{
  "companies": "Acme Corp, Globex, Initech",
  "focus": "AI product strategy in 2025"
}
```

You should see **three** child runs under one parent **[session](/product/concepts/session)** when the model issues three sub-agent **operation** calls in parallel (names in the UI may match your Sub-agent label). If you only see one child call, check **Execution model** is **Parallel** and the instructions forbid batching companies into a single payload.

---

## Step 4 — Test the parent

1. **Test run** the **parent**—use the sample JSON above for **Parallel**, or your own input that forces multiple same-sub-agent calls in one turn.
2. In the **[run](/product/concepts/run)** inspector, confirm child runs appear under the same **[session](/product/concepts/session)** and that token totals roll up as expected.

---

## Step 5 — Deploy the parent

1. **Deploy** the parent’s testing snapshot when satisfied.
2. If you use **[triggers](/product/concepts/trigger)**, remember automation runs the **live** parent version only.

---

## Common pitfalls

| Issue                       | What to check                                                                                             |
| --------------------------- | --------------------------------------------------------------------------------------------------------- |
| **Multi-agent** tab missing | Plan feature; agent must be **Request** or **HITL**, not **Sub-agent**.                                   |
| Cannot add a sub-agent      | Target is not **Sub-agent** mode, self-reference, over allow-list cap, or **Can execute sub-agents** off. |
| **Save** fails              | **Sub-agent** **description** empty; invalid **Anomaly control (L1)** rules on **[Agent version](/product/concepts/agent-version)**; plan depth. |
| Delegation fails at runtime | **Sub-agent** not **live**; **session** or **depth** limits exceeded.                                     |
