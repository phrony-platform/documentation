title: "Tutorial: Multi-agent system with the CLI and TypeScript SDK"
description: "Define a parent agent and sub-agents in YAML with the Phrony CLI, apply to your workspace, then start runs from your own backend with @phrony/sdk."
---

This guide walks you from **manifest as code** to a **working HTTP integration**: you model a small [multi-agent system](/techniques/multi-agent-systems) in YAML, reconcile it with **`phrony apply`**, then drive the **root** agent from your service using the [**TypeScript SDK**](/packages/typescript-sdk) (`@phrony/sdk`).

You should already have a [workspace](/concepts/workspace), an [LLM provider](/concepts/llm-provider) named in Phrony, a plan that supports **multi-agent** delegation, and a **Telegram** integration if you apply the manifest below (same integration pattern as the [Telegram + public API sample](/samples/manifests#telegram-public-api-sample)).

## What you will build

| Layer | What you do |
| ----- | ----------- |
| **Manifest** | One [manifest](/concepts/manifest) document shaped like the [Telegram + public API sample](/samples/manifests#telegram-public-api-sample): **Cat Facts** HTTP and **Telegram** (send requires approval). A **parent** agent (`executionMode: hitl`) owns the **API** trigger and Telegram send, with `canExecuteSubAgents` and an **allowlist**; a **sub-agent** (`executionMode: sub_agent`) only **fetches** the fact and returns structured JSON. |
| **CLI** | `phrony init` (optional), edit YAML, `phrony lint`, `phrony login`, `phrony plan`, `phrony apply`. |
| **Your system** | Create an **API key** scoped to the parent’s **API** trigger, then call `startRun` / `getRun` (and optionally `getConversation`) from your backend. |

The **API** trigger lives on the **parent** only. You never start runs directly on sub-agents when they are in **Sub-agent** mode; the parent delegates inside one [session](/concepts/session).

## Part 1 — Scaffold and author the manifest

### 1. Create a project folder (optional)

If you do not already have a manifest repo, scaffold one:

```bash
mkdir phrony-multi-agent-demo && cd phrony-multi-agent-demo
pnpm dlx @phrony/cli init
```

That creates `manifests/`, `phrony.config.json`, and a starter file. You can instead add `manifests/tutorial.yaml` to an existing repo.

See **[phrony](/packages/cli)** for global flags, credentials, and CI patterns.

### 2. Point the CLI at your workspace

Edit **`phrony.config.json`** (or use environment variables) so network commands know your tenant and API origin:

| Field | Where to get it |
| ----- | ---------------- |
| `tenantId` | Workspace / tenant id from the Phrony dashboard |
| `apiBase` | Usually `https://api.phrony.com` unless your team uses a custom gateway |

### 3. Replace the example with a two-agent manifest

Save the following as **`manifests/tutorial.yaml`** (or merge into your index). It reuses the **inputs**, **services**, and **HITL Telegram send** pattern from [`telegram-public-api-notify.yaml`](../samples/manifests/telegram-public-api-notify/manifests/telegram-public-api-notify.yaml), but splits **fetch** (sub-agent, public HTTP only) from **send** (parent, Telegram only).

Add **`manifests/phrony.values.yaml`** beside it using the same **`inputs`** keys as **`telegram-public-api-notify/manifests/phrony.values.yaml`** in the docs sample (see [Telegram + public API sample](/samples/manifests#telegram-public-api-sample); adjust **`telegram_chat_id`**, **`telegram_service_name`**, **`llm_provider_name`**, and **`llm_model`** for your workspace).

```yaml
kind: phrony.manifest
version: 1
metadata:
  label: multi-agent-cli-sdk-tutorial
  rootManifestKey: telegram_public_api_parent
inputs:
  - key: llm_provider_name
    type: llm_provider
    description: Name of an LLM provider already configured in the target workspace.
  - key: llm_model
    type: string
    description: Model id for the chosen provider (for example gpt-4o-mini).
  - key: telegram_service_name
    type: integration
    description: Existing Telegram integration this sample should use (the one already listed under Integrations).
  - key: telegram_chat_id
    type: string
    description: Telegram chat id your bot should send to (digits, or a negative id for a group or supergroup).
  - key: agent_display_name
    type: string
    description: Human-readable agent name in the dashboard (parent); the fetcher appears as this name plus a suffix.
  - key: api_trigger_name
    type: string
    description: Label for the API trigger your systems call to start a run.
llmProviders:
  - name: "{{inputs.llm_provider_name}}"
    type: openai
services:
  - manifestKey: svc_public_catfacts_http
    name: "Cat Facts (public HTTP)"
    type: http
    config:
      baseUrl: "https://catfact.ninja"
      auth:
        method: none
    operations:
      - manifestKey: op_fetch_cat_fact
        name: fetch_cat_fact
        description: GET a random cat fact from the public API (JSON body includes a fact string).
        configuration:
          method: GET
          path: /fact
  - manifestKey: svc_telegram_builtin
    name: "{{inputs.telegram_service_name}}"
    type: built_in
    config:
      toolkitSlug: TELEGRAM
      authScheme: API_KEY
    operations:
      - manifestKey: op_send_telegram_message
        name: send_telegram_message
        description: Send plain text to a Telegram chat; pass chat_id (use the id from the agent instructions).
        configuration:
          toolSlug: TELEGRAM_SEND_MESSAGE
          toolVersion: latest
        requireApproval: true
agents:
  - manifestKey: telegram_public_api_fetcher
    name: "{{inputs.agent_display_name}} — cat fact fetcher"
    executionMode: sub_agent
    llmProviderName: "{{inputs.llm_provider_name}}"
  - manifestKey: telegram_public_api_parent
    name: "{{inputs.agent_display_name}}"
    executionMode: hitl
    llmProviderName: "{{inputs.llm_provider_name}}"
versions:
  - agentManifestKey: telegram_public_api_fetcher
    status: testing
    versionLabel: v1
    llmModel: "{{inputs.llm_model}}"
    description: Fetches one cat fact from the public HTTP API; invoked only as a sub-agent tool from the parent.
    instructions: |
      You are the fetch-only step in a Telegram relay. Call **fetch_cat_fact** exactly once.
      Return JSON that matches your output schema: put the API fact string in `fact`. Echo `contextHint` from your input when provided (otherwise use an empty string).
      Do not call fetch_cat_fact more than once. Do not send Telegram messages.
    temperature: 0.25
    maxIterations: 8
    maxToolCalls: 4
    allowedOperations:
      - serviceName: "Cat Facts (public HTTP)"
        operationName: fetch_cat_fact
    inputSchema:
      type: object
      properties:
        contextHint:
          type: string
          description: Optional short line from the parent (for example from the API trigger payload)
    outputSchema:
      type: object
      properties:
        fact:
          type: string
        contextHint:
          type: string
      required: [fact]
  - agentManifestKey: telegram_public_api_parent
    status: testing
    versionLabel: v1
    llmModel: "{{inputs.llm_model}}"
    description: Orchestrates the relay—delegates fetch to the sub-agent, then sends one Telegram message (HITL approval on send).
    instructions: |
      Your job each run: put **one** cat fact from the sub-agent into **one** Telegram message.

      Do this in order, once per run:
      1. Call the **telegram_public_api_fetcher** sub-agent once. Pass `contextHint` from the API trigger payload field **`note`** when it is a non-empty string; otherwise pass an empty `contextHint`.
      2. Build the Telegram body from the child's `fact` (plain text). You may add **at most one** short prefix line only if `contextHint` or `note` gives useful context (for example an alert id); keep the fact the main content.
      3. Call **send_telegram_message** with **chat_id** **{{inputs.telegram_chat_id}}** and your composed message. Use any other required argument names exactly as your Telegram tool defines them.

      Do **not** ask what text to send—the fact comes from the child. Do not call the fetcher more than once.

      If the operator explicitly gives a **different** chat id in this conversation and clearly confirms using it for this run, use that id instead of **{{inputs.telegram_chat_id}}** for send_telegram_message only.

      Do not claim the message was sent until the Telegram tool succeeds.
    temperature: 0.25
    maxIterations: 12
    maxToolCalls: 8
    canExecuteSubAgents: true
    subAgentExecutionModel: sequential
    allowedSubAgents:
      - telegram_public_api_fetcher
    allowedOperations:
      - serviceName: "{{inputs.telegram_service_name}}"
        operationName: send_telegram_message
    inputSchema:
      type: object
      properties:
        note:
          type: string
          description: Optional short line from your API caller surfaced as context for the fetcher prefix
    outputSchema:
      type: object
      properties:
        summary:
          type: string
      required: [summary]
triggers:
  - agentManifestKey: telegram_public_api_parent
    name: "{{inputs.api_trigger_name}}"
    type: api
    rpmLimit: 60
    maxConcurrentRuns: 4
    maxInputTokens: 16000
    exposeStepTimelineToApi: true
```

Design notes (see **[Multi-agent systems](/techniques/multi-agent-systems)** for the full model):

- **`executionMode: sub_agent`** on the fetcher makes it callable only as a **sub-agent tool** from an allowlisted parent; it has **no** Telegram tools—only **`fetch_cat_fact`**.
- **`executionMode: hitl`** on the parent matches the sample: **require approval** on **`send_telegram_message`** surfaces in **Review** before delivery.
- **`canExecuteSubAgents: true`** and **`allowedSubAgents`** gate which children the parent’s model may invoke.
- **`subAgentExecutionModel: sequential`** runs one child at a time in order; use **`parallel`** when you want batched child runs.
- **`inputSchema` / `outputSchema`** on the **sub-agent** keep the fetch handoff predictable; the parent adds **`note`** / **`summary`** for API callers and completion output.

If your **`manifests/index.yaml`** still includes the stock `example.yaml`, either remove that include or ensure labels and keys do not conflict.

### 4. Lint, sign in, plan, and apply

```bash
pnpm dlx @phrony/cli lint manifests/tutorial.yaml
pnpm dlx @phrony/cli login
pnpm dlx @phrony/cli plan manifests/tutorial.yaml
pnpm dlx @phrony/cli apply manifests/tutorial.yaml
```

`apply` runs a dry run first, then prompts for confirmation unless you pass **`--auto-approve`** (for scripts). Treat the plan output as your review gate before reconciliation.

After a successful apply, note the **root agent id** in the CLI result if your CLI prints it, or open **Agents** in the Phrony dashboard and find the **parent** agent (the name you set in **`agent_display_name`** in **`phrony.values.yaml`**)—you need its **agent id** (UUID) for the SDK. On the agent’s **Triggers** page, open the **API** trigger you declared; you need that **trigger id** when you scope an API key.

## Part 2 — API access from your backend

### 1. Create an API key

In the Phrony dashboard: **Settings** → **API keys** (or your workspace equivalent).

- Create a key with prefix **`phk_`** and store it in a **secret manager** or environment variable (`PHRONY_API_KEY`). Never ship it to a browser.
- Add a **scope** that includes the **parent** agent and its **API** trigger only.

The agent’s **Access** tab shows **`agentId`** and the public **base URL**—match those to `AGENT_ID` and `PHRONY_API_BASE` in your code.

### 2. Call shape

Runs always target the **parent** agent id. `input` must satisfy the parent’s active version **`inputSchema`**—here you may pass `{ "note": "..." }` for an optional prefix line, `{}`, or omit optional fields per your schema.

For **HITL**, streaming, or **user task** completion over HTTP, reuse the patterns in **[Example: Building an embedded agent](/academy/example-embedded-agent)** (expose timeline on the API trigger when you need conversation steps from the API).

## Part 3 — Integrate with `@phrony/sdk`

Install the SDK in **your** service (backend or worker—not a public client bundle):

```bash
pnpm add @phrony/sdk
```

Minimal **start + poll** loop:

```typescript
import { Phrony } from "@phrony/sdk";

const phrony = new Phrony({
  apiKey: process.env.PHRONY_API_KEY!,
  baseUrl: process.env.PHRONY_API_BASE ?? "https://api.phrony.com",
});

const agentId = process.env.AGENT_ID!;

const { runId, status } = await phrony.startRun(agentId, {
  input: {
    note: "Backend demo — JWST",
  },
});

let run = await phrony.getRun(runId);
while (shouldKeepPolling(run.status)) {
  await sleep(1500);
  run = await phrony.getRun(runId);
}

console.log("final status:", run.status);
console.log("output:", run.output);

function shouldKeepPolling(status: string) {
  // Exact strings depend on your workspace; align with getRun docs / dashboard.
  return status === "Running" || status.startsWith("Waiting");
}

function sleep(ms: number) {
  return new Promise((r) => setTimeout(r, ms));
}
```

**Observability:** Use **`getConversation(runId)`** to inspect merged timeline **`items`**—you should see **sub-agent** steps on the parent run and child activity in the same [session](/concepts/session). For live updates without polling, use **`streamRunEvents`** from a server environment (see **[TypeScript SDK](/packages/typescript-sdk)**).

**Wiring into your product:** keep `Phrony` construction and `startRun` inside your API layer (Express route, Next.js server action, queue consumer, and so on). Map your automation’s context into the optional **`note`** field (or pass **`{}`**), persist **`runId`** if you need async completion, and return **`run.output`** when `status` is **Completed**.

## Checklist

- [ ] **`llmProviders`** name matches a configured provider in your workspace (via **`llm_provider_name`** in **`phrony.values.yaml`**)
- [ ] Sub-agent **`executionMode: sub_agent`** (fetcher); parent **`executionMode: hitl`** so Telegram **require approval** queues in **Review**
- [ ] Parent version has **`canExecuteSubAgents: true`**, **`allowedSubAgents`** listing each child’s **`manifestKey`**, and a **`subAgentExecutionModel`**
- [ ] **Telegram** integration connected; **`telegram_chat_id`** and **`telegram_service_name`** set for apply
- [ ] Both agents have **testing** or **deployed** versions before you rely on production traffic
- [ ] **API** trigger on the **parent**; API key scoped to that agent + trigger
- [ ] **`PHRONY_API_KEY`**, **`AGENT_ID`**, and optional **`PHRONY_API_BASE`** set in the integration environment
