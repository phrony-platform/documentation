You can **dry run** and **apply** manifests from the **Phrony dashboard** or with the commands on this page. Dashboard flows use **Import manifest** (agents list) or **Apply manifest** (**Agent options** on an agent detail page): upload YAML, complete **checks**, map **inputs**, then **dry run** or **apply**. See [Manifest — Where you use manifests in Phrony](/concepts/manifest#where-you-use-manifests-in-phrony).

### With the CLI

Network manifest commands use **`Authorization: Bearer`** with **`phrony login`** or **`PHRONY_ACCESS_TOKEN`**. They call the same Phrony API endpoints as other CLI network commands (not the public **`/v1`** surface). See [Authentication](/packages/cli/authentication).

**`phrony plan`**, **`phrony apply`**, and **`phrony diff`** accept a **file** or a **directory** that contains `index.yaml` / `index.yml`. Default path: `./manifests`.

Validate YAML first with **`phrony lint`** (see [Init and lint](/packages/cli/init-and-lint)); pass **`--values`** when manifests declare inputs.

For parameterized manifests, put a **`phrony.values.yaml`** next to the manifest entry (or pass **`--values`** / set **`PHRONY_MANIFEST_VALUES`** to a file path). The file must declare `version: 1` and an `inputs` mapping (string keys to scalar values). See [Manifest — Parameterized values](/concepts/manifest#manifest-inputs). In the Phrony dashboard, you supply the same logical **inputs** in the **Import manifest** or **Apply manifest** dialog instead of a values file.

## `phrony plan [path]`

**Dry-run** manifest apply: sends merged YAML to the API with `dryRun=true` and prints a planned-change summary (table by default).

**In the Phrony dashboard:** after you open **Import manifest** or **Apply manifest** and upload YAML, choose **dry run** when the dialog offers it—same idea as a non-mutating plan before you commit changes.

| Flag | Description |
| ---- | ----------- |
| `--tenant <id>` | Workspace tenant id (overrides env / config). |
| `--prune` | Pass `prune=true` to the apply endpoint. |
| `--name-suffix <s>` | Optional `nameSuffix` query parameter. |
| `--anchor-agent <uuid>` | Optional `anchorAgentId` query parameter (scopes reconciliation when supported). |
| `--values <path>` | Path to `phrony.values.yaml` (overrides `PHRONY_MANIFEST_VALUES` and auto-discovery beside the manifest). |

---

## `phrony apply [path]`

Runs **plan** (dry-run), then prompts **Apply these changes? [y/N]** unless you pass **`--auto-approve`**.

| Flag | Description |
| ---- | ----------- |
| Same as `plan` | `--tenant`, `--prune`, `--name-suffix`, `--anchor-agent`, `--values` |
| `--auto-approve` | Skip the interactive confirmation (required for non-interactive applies). |

**JSON behavior:** With `--json` and **without** `--auto-approve`, `apply` only outputs the dry-run result and does **not** perform the mutating apply (use `--auto-approve` with `--json` for scripting).

**In the Phrony dashboard:** use **apply** in the same dialog after you review the dry-run result (the CLI’s confirmation prompt is your local equivalent of that step).

---

## `phrony diff [path]`

Compares your **local** resolved manifest to the manifest **exported** from the cloud for a given **root agent**.

You must supply the root agent id for the export side via **`--agent`**, **`PHRONY_ROOT_AGENT_ID`**, or `rootAgentId` in `phrony.config.json`.

| Flag | Description |
| ---- | ----------- |
| `--tenant <id>` | Workspace tenant id (overrides env / config). |
| `--agent <uuid>` | Root agent id for export (overrides env / config). |

**In the Phrony dashboard:** there is no single command that matches **`phrony diff`**. Use **Export manifest** (agents list, **Agent options**, or the **Manifest** sub-tab on an agent) to download YAML from the cloud, then compare files in your editor or source control. **`phrony diff`** automates that comparison for CLI workflows.
