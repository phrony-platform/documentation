Phrony exposes two different **programmatic** credential types. They are **not** interchangeable:

- **API keys** (`phk_…`) — send **`X-API-Key`**. For **[public `/v1` routes](/api-reference/index)** such as runs, sessions, triggers, and file library when the key allows it. Narrow, external-style automation.
- **Workspace access tokens** (`pwt_…`) — send **`Authorization: Bearer`**. For **internal tenant APIs** on the gateway (for example manifest operations, members, billing surfaces the token is scoped for). Created in the Phrony dashboard; use **`PHRONY_ACCESS_TOKEN`** with the **[CLI](/packages/cli)** for CI. There is **no** migration from API keys to access tokens unless you explicitly create new tokens.

Interactive people use the **Phrony dashboard** (session cookies / JWT) or **`phrony login`** (OAuth) instead of storing long-lived API keys on a laptop when possible.

The sections below focus on **API keys** and the public JSON API. Workspace access tokens follow the same Bearer header shape as dashboard sessions but are documented with the **[CLI](/packages/cli)** and dashboard copy where you create them.

## API keys

> Create and revoke keys in the **Phrony dashboard** under **Settings → API keys**. Each key is a single secret; issued keys are prefixed with **`phk_`**. Do not commit keys to source control or put them in client-side code; use environment variables or a secret manager on your server. These keys do **not** grant arbitrary internal administration; for that, use **[workspace access tokens](/packages/cli/authentication#workspace-access-tokens-ci)** (or sign in as a person).

## Headers (authenticated routes)

| Header | Required | Value |
| ------ | -------- | ----- |
| `X-API-Key` | Yes for authenticated routes | Your workspace API key (prefix `phk_`). |
| `Content-Type` | When sending a JSON body | `application/json` |

Example:

```http
X-API-Key: <your key>
Content-Type: application/json
```

The gateway allows `X-API-Key` in CORS preflight.

### Example

Every authenticated request adds the same header. Replace `RUN_ID` after you [start a run](/api-reference/agents#start-a-run).

<CodeGroup>

```bash cURL
export PHRONY_API_BASE="${PHRONY_API_BASE:-https://api.phrony.com}"
export PHRONY_API_KEY="phk_..."

curl -sS "${PHRONY_API_BASE}/v1/runs/00000000-0000-0000-0000-000000000001" \
  -H "X-API-Key: ${PHRONY_API_KEY}"
```

```typescript TypeScript
async function example() {
  const baseUrl = process.env.PHRONY_API_BASE ?? "https://api.phrony.com";
  const apiKey = process.env.PHRONY_API_KEY ?? "";
  const runId = "00000000-0000-0000-0000-000000000001";
  const res = await fetch(`${baseUrl}/v1/runs/${runId}`, {
    headers: { "X-API-Key": apiKey },
  });
  console.log(res.status, await res.json());
}
void example();
```

```python Python (stdlib)
import json
import os
import urllib.request

BASE = os.environ.get("PHRONY_API_BASE", "https://api.phrony.com").rstrip("/")
req = urllib.request.Request(
    f"{BASE}/v1/runs/00000000-0000-0000-0000-000000000001",
    headers={"X-API-Key": os.environ["PHRONY_API_KEY"]},
)
with urllib.request.urlopen(req) as res:
    print(json.load(res))
```

</CodeGroup>

## Scoping

Keys are **scoped to your workspace** and to specific **agent** and **API trigger** pairs (and optionally other constraints). The scope determines what you may call:

| Your call | What Phrony checks |
| --------- | ------------------ |
| `POST /v1/agents/{agentId}/runs` | The key must be scoped to **that** `agentId` and the **API** trigger used to start the run. The path `agentId` must match. |
| `GET /v1/agents/{agentId}/sessions` | Same as above: `agentId` in the path must match the key’s agent scope. |
| `GET` / `POST` under `/v1/runs/{runId}/…` | The key must be an **active** key for the workspace, and the key must be **allowed to access that run** (same agent, and when the run is tied to a trigger, a scope that includes that trigger). |
| `POST /v1/file-library/presign` and `…/finalize` | The key must have **file uploads** enabled. |

If a key is missing, wrong, or out of scope, the API returns **401** (missing or invalid key) or **403** (forbidden) as appropriate.

## File uploads

`POST` to `/v1/file-library/*` returns **403** if **Allow file uploads** is not enabled for that key, even when the key is otherwise valid.

See **[File library](/api-reference/file-library)** for the upload flow.
